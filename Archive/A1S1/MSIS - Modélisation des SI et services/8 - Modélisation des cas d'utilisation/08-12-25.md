## Modélisation des cas d'utilisation

Méthode OOSE - Object Oriented Software Engineering : Méthode OO fondée sur cas d'utilisation
### OOSE - Présentation
#### Objectifs :
- Définir périmètre fonctionnel 
	- Fonctionnalités : Ce qu'on peut faire
- Spécifier comportement
	- Vu par une utilisateurice
	- ex : acheter billet de concert -> qu'est ce que je vois si je clique sur le bouton réserver
- Ident. et comprendre besoins fonctionnels des utilisateurs
	- De quoi avez vous besoin?

#### Principes de base : 
- Modéliser du pdv des users != implémentation profonde
	- Comment diff. users imaginent usage du système
	- Raisons utilisation (peuvent être variées)
	- Comportement **de l'utilisateurice** face au système
- Décrire ce que syst. doit faire (comportement souhaité **par utilisateurice**)
	- Pas question d'implémentation/comment réaliser ce comport.
	- Pas de détails de dev
	- Indépendant de la technologie utilisée

#### Concepts :
1. Acteur : Rôle joué par utilisateurice humaine (ou parfois machine/automate) qui interagit avec syst.
		ex : Etudiante
2. Cas d'utilisation : Objectifs fonctionnels des acteurs - ce qu'un acteur devrait être capable de faire avec le système
		ex : S'inscrire à un cours
### Concepts
#### Acteur :
- = tout ce qui existe en dehors du système logiciel (entre la chaise et le clavier) et interagit directement avec le système -> échange de l'info
	- Utilisateurices humaines
	- Machines/Systèmes pairs
	- Organisations
- Type ou catégorie d'utilisateurice du syst.
- Rôle générique joué par user dans un contexte d'usage du syst
	- m^m personne peut jouer plusieurs rôles et instancier plusieurs acteurs
##### Identification des acteurs :
- Utilisateurices potentielles du système de la bibliothèque?
	- Abonné, Bibliothécaire
- Qui assure maintenance du système?
	- Admin
- Quels systèmes en relation avec système de bibliothèque?
	- Système du réseau des bibliothèques
#### Cas d'utilisation (use case)
- = Description d'un ensemble de séquences d'actions incluant variantes, qu'un syst. effectue pour fournir résultat et avec valeur pour acteur
	- Décrit utilisation syst. par acteur pour raison particulière pour satisfaire objectif
	- Décrit interaction entre acteur et système pour satisfaire cet objectif
	- Suite complète d'actions acteur <-> système
##### Identification des use cases
- Identification par biais des acteurs :
	- à chaque suite d'évènements initiée par acteur, on associe use case : En considérant chaque acteur, décision quel cas utilisation pour quel acteur
- On pose des questions pour chaque acteur
	- Principales tâches acteur
	- Raisons utilisation syst pour acteur
	- Ecrire/Lire/Modifier?
	- Informer syst de modif extérieures?
	- acteur = souhaite informé de modifications?
##### Mais - sélection (cas de potentiels doublons) : 
- ex. Bibliothèque : 
	- Abonné s'inscrire <=?> Bibliothécaire enregistrer abonné?
	- Abonné emprunter <=?> Bibliothécaire enregistrer emprunt?
=> Overlap

##### Rédaction des use cases  :
#### Scénarios
Chaque modèle peut à priori être écrit avec 1 scénario de base (réussite) et n scénarios d'exception (toutes les variantes exceptionnelles). Aujourd'hui plus complexe car systèmes == très complexes.

ex : Enregistrer emprunt 

><u>Scénario de base : </u>
>1. La bibliothécaire sélectionne la fonctionnalité ‘Enregistrer un emprunt’. 
>2. Le système demande de scanner la carte de l’abonné. 
>3. La bibliothécaire scanne la carte de l’abonné. 
>4. Le système identifie l'abonné et affiche l’information concernant les emprunts en cours et l’état de l’abonné. 
>5. Si l’abonné est actif est le nombre d’emprunts est inférieur au nombre maximal permis le système demande de scanner le code barre de l’ouvrage. 
>6. La bibliothécaire scanne le code barre de l’ouvrage.
>7. Le système affiche le message ‘Emprunt enregistré’ et le nouvel emprunt est ajouté dans la liste des emprunts de l’abonné. Scénarios d’exception :Le nombre d’emprunts autorisé est atteint 
>5.1 Si l’abonné a déjà le nombre maximal d’emprunts, le système affiche le message ‘Interdiction d’emprunter d’autres ouvrages’

##### Règles d'écriture de scénarios : 
- Décrire : 
	- Quoi, pas comment
		- Vérifier l'identité user
		- !utiliser un formulaire avec un call db
- Autonomie :
	- Ne pas fusionner plusieurs cas
		- Gérer catalogue = abstrait
		- Ajouter produit, modifier produit, retirer produit
- Scénario = transaction
	- Début, fin
	- Exécuté complètement ou pas du tout
- N° de ref/id pour chaque action
- Phrases simples et claires
- Style direct : 
	- Pas d'ambigüités ('très', "plutôt", "peu" ...)
	- Pas de références métonymiques comme 'il' 'lui', 'leur'...
	- Eviter synonymes et homonymes

### Structuration du modèle de cas d'utilisation
Extension, Inclusion, Abstraction

#### Association d'extension :
Permet d'étendre un cas d'utilisation déjà complet pour modéliser : 
- Parties optionnelles du cas
- Scénarios rares de remplacement
- Sous scénarios
--> Optionnel et rare
Formalisation :
- Point d'extension - définit endroit dans cas principal où cas peut être inséré
	- \<nom> - chaque pt a un nom
	- \<endroit> - description optionnelle pour indiquer endroit où cas d'extension peut être inséré
- Condition d'extension - précise condition d'exécution
