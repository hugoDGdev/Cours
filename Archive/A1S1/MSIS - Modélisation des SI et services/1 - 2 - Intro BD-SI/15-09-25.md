Def SI : Ens organisé resources et dispositifs -> collecter, stocker, traiter, diffuser infos nécessaires fonctionnement orga (admin, entreprise, institution...)

------------------------------
Exemple : Entreprise transports publics

- Besoin SI pour gérer réseau
- lignes, véhicules, chauffeurs...

Problèmes : 
	Basiques :
		- Chauffeur en congé le 30
		- Ligne 2 le 31 avec bus 52
		- Qui a assuré L18 le 15/03 16h-18h?
		- Véhicule 45 en marche ajd?
		- 1 véhicule => même ligne?
	Critiques : 
		- Chauffeur malade; est ce que et comment remplacer? 
		- Tram 12 bloqué par accident -> solution secours
	Evènements uniques et/ou récurrents : 
		- Retirer véhicule 124 de circulation
		- Mettre nouvelle ligne en circulation 
	Respect des règles de gestion (possibilité d'évoluer):
		- Verif permis valable pour bon type de véhicule par chauffeur
		- Plusieurs types de véhicules par chauffeur
	
Solutions : 
	Exemple : gestion du planning des lignes : 
		- Solution 1 : Stockage fichier/tableau :
			- > Chaque ligne = affectation chauffeur et véhicule sur 1 ligne 1j donné
			Problèmes : 
				- Redondance des infos (coordonnées chauffeurs… etc.)
				- Taille du fichier
				- Complexité de consultation
				- Cohérence + consistance en cas de modification/ajout
		- Solution 2 : Stockage multitableau :
			- > Un fichier pour chaque catégorie de données (chauffeurs, véhicules… etc.)
			Problèmes résolus par rapport à S1 : 
				- Pas de redondance
				- Pas de problème de màj -> cohérence assurée
			Problème :
				- Recherche complexe, connexion manuelle entre tables
		- Solution 3 : Du fichier à la BD
			- Pas de redondance
			- màj facile et continue
			- data facilement accessible
			- indépendance data/traitements
			- vision globale

BD ?

Def : ens structuré de data sur supports info permettant satisfaire simultanément plusieurs clients

Utilités :
	- Gros volumes de données (+ que nb de traitements et structure data + que struc. algo.)
	- Partage infos plusieurs personnes/entreprise
	- Gestion de l'accès aux infos
	- Assurer cohérence et non redondance data (data persistante)
Infos à registerer : 
	- Faits
	- Propriétés/règles
	- Evenements
Approche de BD : 
	- Description des data par modèles/schémas de données (conceptuel)
	- Validation (normalisation) par règles du schéma
	- Réalisation pratique avec SGBD (Syst. de gestion de BD)
	- Programmation accès data avec 1/a langage.
Caractéristiques de cette approche : 
	- Abstraction data (représentation conceptuelle != physique)
	- Indépendance données/programme (programme accès écrit != stockage) => changement de data n'invalide pas le programme
	- [[Schéma vues multiples.canvas|Vues multiples]] (plusieurs types de vues data en fonction de l'utilisateur)
Défis :
	- Stockage : 
		- Quelles données stockées?
		- Sous quelle forme?
		- Structure
